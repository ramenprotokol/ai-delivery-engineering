# AI Code Review Checklist

Use this checklist when reviewing any code that was generated, completed, or substantially modified by an AI tool (Claude Code, Cursor, GitHub Copilot, or similar). It supplements — it does not replace — a standard code review.

> **The reviewer owns this code.** Merging AI-generated code without this review is the same as merging unreviewed code. The AI tool that wrote it cannot be held accountable. The reviewer can.
>
> **AI-assisted. Human-reviewed. Human-approved.** This checklist is how "human-reviewed" becomes a real claim rather than a rubber stamp.

---

## 1. Existence and Accuracy of References

- [ ] Every function, method, and class the AI called or imported actually exists in the stated library or SDK — verified against official documentation, not assumed from the model's output
- [ ] Every API endpoint, field name, or protocol the AI referenced has been confirmed against current API documentation (AI models have knowledge cutoffs and regularly hallucinate deprecated or invented APIs)
- [ ] Library versions used match the versions in the project's dependency file — the AI may have generated code for a different version than what is installed
- [ ] If the AI cited a source, example, or reference, that source exists and says what the AI claimed

## 2. Logic and Intent

- [ ] The generated code does what the ticket, spec, or prompt intended — not just what the AI inferred
- [ ] The logic has been read top to bottom by a human who can explain what each section does in plain English
- [ ] Control flow (conditionals, loops, early returns) matches the intended behavior — the AI sometimes inverts conditions or off-by-ones at boundaries
- [ ] Data transformations produce the expected output — trace at least one realistic input through the code manually
- [ ] Any assumptions the AI made (about input shape, caller behavior, environment) are documented and validated

## 3. Edge Cases and Error Handling

- [ ] Null, empty, and zero-value inputs are handled correctly — not silently passed through or crashed on
- [ ] Error paths are explicit — the code fails loudly with a useful message rather than swallowing exceptions or returning misleading defaults
- [ ] Timeouts, retries, and rate limits are addressed wherever the code calls an external service
- [ ] Concurrency issues (race conditions, shared mutable state) have been considered if the code runs in a concurrent context
- [ ] Resource cleanup (file handles, connections, locks) happens on both the happy path and error paths

## 4. Security

- [ ] No secrets, credentials, tokens, or personally identifying data appear in the generated code or comments
- [ ] All external input (request parameters, file content, user data, environment variables) is validated before use — the AI sometimes skips validation steps
- [ ] No insecure defaults are present: no `verify=False`, no `0.0.0.0` bindings in production configs, no permissive CORS policies, no plaintext storage of sensitive values
- [ ] SQL queries, shell commands, and filesystem paths built from user input use parameterization or sanitization — not string concatenation
- [ ] Authentication and authorization checks are in the right place — not missing, not duplicated in a way that creates a bypass path
- [ ] The [security-review.md](security-review.md) checklist has been run if this code touches auth, secrets management, or user data

## 5. Silent Behavior Changes

- [ ] Existing callers of any modified function or method still receive the same behavior unless a breaking change was intentional and documented
- [ ] Default argument values, return types, and exception types have not changed silently
- [ ] The AI has not removed or weakened existing validation, rate limiting, or access control logic to make the new code simpler
- [ ] Database queries have not changed in a way that would return different rows, columns, or ordering without that being explicitly required

## 6. Tests

- [ ] Tests generated by the AI actually test the behavior — run the code path manually and confirm the test would fail if the implementation were broken
- [ ] Tests do not simply repeat the implementation (e.g., `assert result == some_function(input)` where `some_function` is the code under test)
- [ ] Mocks and stubs reflect realistic behavior of the things they replace — not idealized behavior that hides real failure modes
- [ ] Test coverage reaches the error paths and edge cases identified in section 3, not just the happy path
- [ ] AI-generated test fixtures use realistic but synthetic data — no real personal data, production IDs, or live credentials

## 7. Performance

- [ ] No new N+1 queries, unbounded loops over large datasets, or repeated expensive operations have been introduced
- [ ] Caching behavior (if any) is correct — the AI sometimes adds caching logic that misses invalidation cases
- [ ] Memory allocations in hot paths are reasonable — the AI sometimes generates clear but inefficient code that would not survive production load
- [ ] If performance matters for this change, a benchmark or load test result exists, not just a "seems fast enough" judgment

## 8. License and Attribution

- [ ] If the AI reproduced a substantial block of code from a known open-source project, the license of that code is compatible with this project's license
- [ ] Any copied or adapted code is attributed in a comment if attribution is required by the source license
- [ ] The AI change log ([templates/ai-change-log.md](../templates/ai-change-log.md)) records what was generated, what tool was used, and what was changed during review

## Review sign-off

By checking all items above, the reviewer confirms they have read and understood the AI-generated code and take responsibility for its correctness and safety.
