# Testing Checklist

Use this checklist before merging any change that adds behavior, modifies behavior, or fixes a bug. Tests are the primary mechanism for proving a change does what it claims and does not break what it claims to leave unchanged. A green CI run is necessary but not sufficient — this checklist confirms the tests themselves are meaningful.

Related files: [pre-merge checklist](pre-merge.md) · [AI code review checklist](ai-code-review.md) · [testing strategy doc](../docs/testing-strategy.md) · [test plan template](../templates/test-plan.md)

## 1. Unit Tests

- [ ] Unit tests exist for every new function, method, or module introduced by this change
- [ ] Each test covers a single behavior — tests that validate multiple independent things are split into separate cases
- [ ] Tests assert on specific outputs, not just that "no exception was thrown"
- [ ] Mocks and stubs are used only to isolate the unit under test from I/O and external services — not to avoid writing difficult tests
- [ ] Test names describe the scenario and expected outcome in plain English (e.g., `returns 404 when resource does not exist`, not `test_get_endpoint`)

## 2. Integration Tests

- [ ] Integration tests exist for every boundary between components: API handlers to database, service to external API, queue consumer to business logic
- [ ] Integration tests run against a real (or realistic in-process) instance of the dependency — not a mock that could mask a schema mismatch or protocol error
- [ ] New database queries are covered by at least one integration test that exercises the actual query against a test database
- [ ] Integration tests verify both the happy path and the failure path — what happens when the dependency is unavailable, returns an error, or returns unexpected data

## 3. End-to-End Tests

- [ ] Critical user-facing paths have end-to-end test coverage: the sequence of steps a user or API consumer takes to complete the most important actions in the system
- [ ] E2E tests run against a deployed environment (staging or ephemeral), not just localhost — environment configuration differences are a real source of failures
- [ ] E2E tests are scoped to the minimum path needed to verify the behavior; they do not replicate unit test logic
- [ ] E2E tests have defined pass/fail criteria, not just "it did not crash"

## 4. Regression Tests for Bug Fixes

- [ ] Every bug fix includes a test that would have caught the bug before the fix — the test fails on the pre-fix code and passes on the post-fix code
- [ ] The test is kept permanently (not marked as `@skip` or `@ignore`) so the bug cannot silently reintroduce itself
- [ ] The commit or PR that introduced the bug is identified and referenced in the test comment, so future readers understand the history

## 5. Edge Cases and Error Paths

- [ ] Tests cover empty inputs, null/nil values, and zero-length collections where these are plausible inputs
- [ ] Tests cover boundary values: off-by-one conditions, maximum allowed values, and minimum allowed values
- [ ] Error paths are explicitly tested: invalid inputs return the correct error type and message, not a 500 or a silent failure
- [ ] Concurrency and race conditions are considered: if the code uses goroutines, async/await, threads, or queues, the tests verify correct behavior under concurrent access where relevant

## 6. AI-Written Tests

- [ ] Every test generated by an AI tool has been read and understood by the reviewer — AI-generated tests are not merged on faith
- [ ] AI-generated tests were verified to actually fail when the behavior they describe is absent: the test was run against the pre-implementation code or a deliberately broken version
- [ ] AI-generated tests do not assert on implementation details (internal variable names, private method calls) — they test observable behavior
- [ ] Any AI-generated test that tests the wrong thing, tests nothing useful, or could never fail has been rewritten or removed

## 7. Determinism and Flakiness

- [ ] Tests do not depend on wall-clock time — time-dependent behavior is tested by injecting a controllable clock
- [ ] Tests do not depend on random values without a fixed seed — or random values are not used in assertions at all
- [ ] Tests do not depend on execution order — each test sets up its own state and tears it down, or uses transactions that are rolled back
- [ ] Network calls in tests are either mocked (unit) or made to a stable test environment (integration/E2E) — no tests call production endpoints
- [ ] The test suite was run at least twice and produced the same result both times before merge

## 8. Coverage Quality

- [ ] Coverage is checked, but the target is meaningful coverage — not the maximum percentage achievable by writing trivial assertions
- [ ] Uncovered lines are reviewed: if they are unreachable dead code, the dead code is removed; if they are untested behavior, a test is added or a documented exception is made
- [ ] Coverage does not include test files themselves, generated files, or configuration boilerplate in the count
- [ ] High coverage on low-value code (getters, constants, auto-generated serializers) does not substitute for coverage on high-value logic (validation, state transitions, authorization checks)

## 9. CI Enforcement

- [ ] All tests run in CI on every PR — no tests that only run locally or require manual triggering before merge
- [ ] CI runs tests in an environment that matches production as closely as possible: same OS, same runtime version, same dependency versions
- [ ] Test failures in CI block merge — the branch protection rule is confirmed active, not assumed
- [ ] CI test run time is acceptable: tests that take more than 10 minutes without justification are flagged for optimization
- [ ] Test output in CI is parseable — test results are reported in a standard format (JUnit XML or equivalent) so failures are surfaced clearly, not buried in log output

> A test suite that passes is not proof the system works — it is proof the system behaves as the tests describe. The quality of the tests determines the value of that proof. Review tests with the same rigor as production code.
